# TCP receive/send path на практике: от sliding window до congestion control

Четыре главы, каждая — отдельная, самодостаточная проблема, которую решает
TCP, с рабочим кодом на C#/.NET 9 и вручную протрассированным примером
(без единого запуска компилятора — трассировка была единственным способом
проверки, и это тоже часть материала: как проверять сетевой код рассуждением,
а не только запуском).

## Порядок чтения

1. **Sliding window** — на чём вообще держится идея "не слать больше, чем
   помещается". SND.UNA/NXT/WND, RCV.NXT/WND, арифметика sequence number
   по модулю 2³².
2. **Out-of-order reassembly** — что делать, когда байты приходят не по
   порядку: как превратить кучу разрозненных диапазонов обратно в
   непрерывный поток, ARM overlap, дубликаты.
3. **Delayed ACK и retransmission timer** — как получатель решает, когда
   ACK-ить, и как отправитель решает, когда считать сегмент потерянным.
   RFC 6298 (Jacobson/Karn), RFC 1122, fast retransmit.
4. **Congestion control** — RFC 5681: slow start, congestion avoidance,
   fast recovery. Почему usable window — это `min(cwnd, rwnd)`, а не
   просто то, что рекламирует получатель.

Каждая глава может читаться отдельно — код в ней самодостаточен и не
зависит от кода последующих глав (хотя главы 2–4 физически лежат в одном
проекте `TcpReceivePipeline`, поскольку демонстрируют работу вместе).

## Карта RFC

| Механизм | RFC | Где в коде |
|---|---|---|
| Формат заголовка TCP, sequence space | RFC 9293 §3.1, §3.4 | Гл. 1 |
| Sliding window (SND/RCV) | RFC 9293 §3.3.1 | Гл. 1 |
| Reassembly, overlap handling | RFC 9293 §3.4 | Гл. 2 |
| Delayed ACK | RFC 1122 §4.2.3.2 | Гл. 3 |
| RTO estimation (Jacobson/Karn) | RFC 6298 | Гл. 3 |
| Fast retransmit | RFC 5681 §3.2 | Гл. 3–4 |
| Slow start / congestion avoidance | RFC 5681 §3.1 | Гл. 4 |

## Сквозной паттерн, который стоит знать заранее

Начиная с главы 3 в коде появляется настоящее время (виртуальные тики,
`PriorityQueue<Action,double>` как очередь событий). Поскольку эта очередь
не умеет отменять произвольный запланированный элемент, везде используется
один и тот же приём: **generation-счётчик**. Каждый раз, когда таймер
логически перезапускается, счётчик увеличивается; сработавший callback
сверяет захваченное им число с текущим — не совпало, событие устарело,
no-op. Увидите это дважды (RTO watchdog и delayed-ACK deadline) — это не
совпадение, а один и тот же приём, примененный к двум независимым таймерам.
