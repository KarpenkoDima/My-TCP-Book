# Глава 1. Sliding window

TCP нумерует не пакеты, а байты потока (RFC 9293 §3.4). Sliding window —
это то, чем стороны договариваются, сколько байт можно держать
"в полёте" одновременно, не дожидаясь подтверждения каждого отдельно.

## Переменные, о которых идёт речь

Каждая сторона соединения держит две пары чисел (RFC 9293 §3.3.1):

```
SND.UNA  — самый старый неподтверждённый байт, который мы отправили
SND.NXT  — следующий байт, который мы отправим
SND.WND  — окно, объявленное нам пиром

RCV.NXT  — следующий байт, который мы ожидаем получить
RCV.WND  — окно, которое мы рекламируем пиру
```

`usable window` — сколько ещё можно отправить прямо сейчас:

```
BytesInFlight = SND.NXT - SND.UNA
UsableWindow  = SND.WND - BytesInFlight
```

## Арифметика по модулю 2³²

Sequence number — это `uint`, и он оборачивается через ноль. Обычное
`a < b` даёт неверный ответ на границе wrap-around. Стандартный приём
(RFC 9293 §3.4, тот же принцип, что и в PAWS) — сравнение через знаковое
вычитание:

```csharp
public static bool LessThan(uint left, uint right)
    => unchecked((int)(left - right)) < 0;
```

Это работает, пока сравниваемые точки находятся на расстоянии меньше 2³¹ —
условие, которое ограниченное окно гарантирует само по себе.

## Код

Файлы в этой главе — самостоятельный проект (`TcpSlidingWindow.csproj`,
компилируется отдельно):

- `Core/SequenceMath.cs` — сравнения по модулю 2³²
- `Core/TcpFlags.cs`, `Core/TcpSegment.cs` — биты заголовка и zero-alloc
  вид на сегмент (`readonly ref struct`, живёт только на стеке одного parse)
- `Core/SlidingWindowTracker.cs` — SND/RCV переменные обеих сторон одного
  соединения в одном объекте (в главах 3–4 это станет проблемой и будет
  разделено — см. врезку ниже)
- `Infrastructure/TcpHeaderParser.cs`, `SyntheticSegmentWriter.cs` —
  парсинг/генерация wire-байт через `BinaryPrimitives`, big-endian,
  без аллокаций
- `Host/Program.cs` — полная передача при узком окне пира: отправитель
  упирается в окно, ждёт ACK, окно открывается

## Пример прогона

MSS=4, окно пира = 12 байт — то есть не больше трёх сегментов
одновременно. Один сегмент намеренно переставлен местами в "сети":

```
SND.UNA / SND.NXT / SND.WND (sender)   RCV.NXT / RCV.WND (receiver)

[step 1] SEND    seq=1001 len=4 payload="ABCD"
[step 2] SEND    seq=1005 len=4 payload="EFGH"
[step 3] SEND    seq=1009 len=4 payload="IJKL"
[step 4] DELIVER seq=1017 len=6 -> OutOfOrder     <- сегмент не по порядку
[step 5] DELIVER seq=1001 len=4 -> InOrder        <- закрывает окно
```

(Полный вывод с числами SND.WND/usable по каждому шагу — в
`Host/Program.cs`, там же комментарии с трассировкой.)

## Врезка: почему этот класс потом разделили

`SlidingWindowTracker` в этой главе хранит **обе** стороны — и SND-, и
RCV-переменные одного соединения — в одном объекте, а `OnSegmentReceived`
просто классифицирует входящий сегмент (`InOrder`/`OutOfOrder`/`Duplicate`)
и на `OutOfOrder` отбрасывает данные. Как только в главе 2 появляется
настоящий reassembly buffer, RCV.NXT перестаёт быть просто полем — он
становится производной от состояния этого буфера. Держать оба источника
истины одновременно — гарантированный баг. В главе 3 это разделено на
`TcpSendWindow` (только SND) и `TcpReceiveEndpoint` (reassembler + честный
RCV.WND). Это не рефакторинг ради стиля — это исправление структурной
ошибки, которую проще было увидеть, уже написав "наивную" версию.
