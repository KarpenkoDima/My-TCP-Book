# Глава 2. Out-of-order reassembly

Sliding window решает "сколько байт может быть в полёте". Reassembly
решает другую задачу: сегменты долетают не в том порядке, в котором были
отправлены (или дублируются, или частично перекрываются) — а приложению
нужен чистый непрерывный поток байтов.

## Задача в одной картинке

```
Пришли сегменты:
SEQ=1000  "Hello "
SEQ=1012  "world!"
SEQ=1006  "TCP "

Порядок получения:   1000 -> 1012 -> 1006
Порядок для приложения: "Hello TCP world!"
```

После первого сегмента `RCV.NXT = 1006`. Сегмент с `SEQ=1012` нельзя
отдать приложению — не хватает байт `1006..1011`. Его нужно временно
сохранить.

## Развёрнутая координата вместо uint напрямую

Хранить `uint` в `SortedDictionary` неудобно из-за wrap-around
(`4294967294 → 4294967295 → 0 → 5` числовая сортировка ломает). Поэтому
внутри reassembler'а используется 64-битная координата, развёрнутая
относительно текущего `RCV.NXT`:

```csharp
int relative = TcpSequence.Distance(RcvNxt, incomingSequence);
long absolute = AbsoluteRcvNxt + relative;
```

Диапазоны хранятся как полуоткрытые интервалы `[Start, End)` — это сильно
упрощает overlap handling.

## Код

Самостоятельный проект (`TcpOutOfOrderReassembly.csproj`):

- `TcpSequence.cs` — та же арифметика по модулю 2³², что и в главе 1
- `BufferedSegment.cs` — один сохранённый диапазон, память через
  `ArrayPool<byte>` (`clearArray: false` — для сетевого reassembly не
  нужно, сегмент просто возвращается в пул как есть)
- `TcpStreamReassembler.cs` — вся логика: приём в любом порядке,
  дедупликация, обрезка перекрытий (`InsertOnlyNewRanges`), каскадная
  выдача (`DrainContiguousData`)
- `Program.cs` — три сценария: обычный reorder + дубликаты, частичный
  overlap, sequence number wrap-around

## Три проверенных сценария

**Reorder.** Финальный поток `"Hello TCP out of order!"`, `RCV.NXT=1023`,
буфер пуст — все диапазоны слились каскадом за один `Push`, когда пришёл
недостающий кусок.

**Overlap.** `[5005,5010) "FGHIJ"` уже сохранён; приходит `[5008,5013)
"IJKLM"` — сохраняются только новые `"KLM"`, `"IJ"` не дублируются.
Итоговый поток: `"ABCDEFGHIJKLM"`.

**Wrap-around.** `initialSequence = uint.MaxValue - 4`. Кусок с
логическим смещением 5 приходит с wire-номером `SEQ=0` (обёртка через
2³²), но внутренняя развёрнутая координата ставит его правильно, после
первого куска — без единой специальной ветки для перехода через ноль.

## Почему не `PriorityQueue<Segment,SequenceNumber>`

Куча (heap) отвечает на вопрос "какой элемент минимальный" — этого
недостаточно. Нужны: поиск всех пересекающихся диапазонов, обрезка
входящего сегмента по границе уже сохранённых данных, поиск
предшественника/преемника. Heap с этим не помогает.
`SortedDictionary<long, BufferedSegment>` даёт упорядоченный по ключу
обход, что и нужно `InsertOnlyNewRanges` для honest overlap-трима.

## Главный инвариант

После каждого `Push`:

1. Все сохранённые диапазоны — справа от `RCV.NXT`.
2. Сохранённые диапазоны не пересекаются друг с другом.
3. В буфере нет уже доставленных байт.
4. Диапазон с `Start == RCV.NXT` немедленно выдаётся.
5. `RCV.NXT` всегда указывает на первый отсутствующий байт.
6. `BufferedBytes` равен сумме длин всех сохранённых диапазонов.

## Правки по внешнему ревью

Черновик этой главы получил разбор со стороны (статический, без
компиляции — как и всё остальное здесь). Разбор нашёл настоящий баг и две
проблемы устойчивости к патологическому трафику. Все три внесены в код:

**1. Потеря данных при исключении в колбэке.** Было:

```csharp
_segments.Remove(start);
_bufferedBytes -= segment.Length;
try { onDataReady(segment.Data); _absoluteRcvNxt += segment.Length; }
finally { segment.Dispose(); }
```

Если `onDataReady` бросает исключение — сегмент уже удалён и уже вернулся
в пул, а `RCV.NXT` не продвинут. Байты потеряны безвозвратно, и главный
инвариант ("RCV.NXT указывает на первый отсутствующий байт") нарушен.
Исправлено — сначала вызов колбэка, удаление и `Dispose()` только после
успешного возврата:

```csharp
onDataReady(segment.Data);
_segments.Remove(start);
_bufferedBytes -= segment.Length;
_absoluteRcvNxt = checked(_absoluteRcvNxt + segment.Length);
segment.Dispose();
```

При исключении сегмент остаётся в буфере и будет предложен заново на
следующем `Push`.

**2. Reentrancy.** "Non-thread-safe" не означает "non-reentrant" — колбэк
теоретически мог вызвать `Push` повторно и получить тот же
ещё-не-удалённый диапазон дважды. Добавлен флаг `_isDelivering`,
`Push` бросает `InvalidOperationException` при повторном входе.

**3. `maxBufferedRanges`.** `receiveWindow` ограничивает суммарные
*полезные* байты, но не число диапазонов — тысячи однобайтовых сегментов
через один создают тысячи узлов дерева и тысячи pooled-массивов
независимо от `receiveWindow`. Добавлен параметр конструктора (по
умолчанию 4096) и новый исход `SegmentInsertResult.ResourceLimitExceeded`.
Компромисс осознанный и задокументирован в коде: проверка грубая — если
буфер уже на пределе, новый сегмент отклоняется целиком, даже если он
закрыл бы существующие gaps и уменьшил бы фрагментацию. Точный расчёт
"каким будет число диапазонов после вставки" стоил бы отдельного прохода
по дереву на каждый `Push`.

Что осталось **не сделано** осознанно (объём для отдельной главы, не для
правки):

- `SortedDictionary` даёт вставку `O(n + k·log n)` вместо `O(log n + k)`
  ожидаемых от дерева с честным `LowerBound` — для по-настоящему
  high-performance версии нужен interval tree или собственный red-black
  tree с поиском первого ключа `≥ x`.
- Соседние диапазоны (`[100,200)` и `[200,300)`) не сливаются в один
  узел — корректно, но не оптимально по числу объектов/pool-арендов.
- `TcpDataReadyHandler` как `void`-колбэк с `ReadOnlySpan<byte>` не
  дружит с `async`, backpressure и сохранением данных за пределы вызова
  — `IBufferWriter<byte>` был бы более гибкой границей API, но это смена
  контракта, а не патч.
- Обнаружение *конфликтующего* overlap (когда новые байты в
  пересечении расходятся со старыми) не реализовано — политика
  "first data wins" молча побеждает без проверки, совпадают ли данные.
- SYN/FIN не входят в API `Push(uint, ReadOnlySpan<byte>, ...)` — сегмент
  с одновременно данными и `FIN` невозможно выразить текущей сигнатурой.
- Автоматических тестов по-прежнему нет — только сценарии в `Program.cs`.
  Минимум для этого компонента: in-order/OOO/duplicate-до-и-после-
  доставки/left-right-overlap/multi-range-cover/window-trim/wrap-around/
  callback-throws/callback-reentrancy/dispose-with-buffered-data/
  fragment-limit, плюс property-based тест (эталонный поток → случайная
  нарезка → перемешивание → сравнение с оригиналом).
