# Глава 3. Delayed ACK и retransmission timer

Главы 1–2 не знали о времени вообще — только о порядке событий. Реальный
TCP должен решить два вопроса, у каждого из которых своя логика и свой
RFC:

- **Получатель**: ACK-ить сейчас или подождать?
- **Отправитель**: когда считать сегмент потерянным?

## Код этой главы строится поверх глав 1–2

`Core/` здесь — это уже не изолированный кусок, а полный интегрированный
приёмный конвейер: `TcpSendWindow` (SND-сторона из главы 1, отделённая от
приёма), `TcpReceiveEndpoint` + `TcpStreamReassembler` (глава 2, приём).
Поверх них добавлены четыре новых файла:

- `InFlightSegment.cs` — один отправленный, но не подтверждённый сегмент.
  В отличие от `TcpSegment` (`readonly ref struct` из главы 1) — обычный
  класс: должен пережить время в полёте и, возможно, ретрансмит; payload
  здесь — копия (`byte[]`), не срез чужого буфера.
- `RetransmissionTimer.cs` — оценка RTO по RFC 6298 (Джейкобсон):

  ```
  SRTT   = (1-α)·SRTT + α·R
  RTTVAR = (1-β)·RTTVAR + β·|SRTT - R|
  RTO    = SRTT + max(G, 4·RTTVAR)
  ```

  и экспоненциальный backoff при реальном истечении таймера (Karn's
  algorithm, RFC 6298 §5.5).
- `RetransmissionController.cs` — очередь неподтверждённых сегментов,
  duplicate-ACK счётчик для fast retransmit (RFC 5681 §3.2: 3 ACK подряд
  с одним и тем же номером), и ключевое правило Карна: **если среди
  подтверждённых одним ACK сегментов есть хоть один ретранслированный —
  RTT-выборка не берётся вообще**, потому что неизвестно, какой из
  передач соответствует этот ACK.
- `DelayedAckPolicy.cs` — RFC 1122 §4.2.3.2: ACK хотя бы на каждый второй
  full-size сегмент, не откладывать дольше фиксированного предела, и
  **немедленный ACK на любой gap** (RFC 5681 §3.2 — иначе отправитель не
  получит duplicate ACK и не сможет сделать fast retransmit).

## Появляется настоящее время — и вместе с ним новая проблема

`Host/Program.cs` в этой главе вводит `PriorityQueue<Action,double>` как
очередь событий вместо раундов. У этой очереди нет операции "отменить
запланированный элемент" — а таймеры нужно перезапускать постоянно
(новый ACK продвинул SND.UNA — RTO пересчитывается; сегмент дошёл не по
порядку — отложенный ACK отменяется в пользу немедленного). Решение —
**generation-счётчик**: при каждом логическом перезапуске счётчик
увеличивается; сработавший callback сверяет захваченное число с текущим
и, если не совпало, тихо ничего не делает. Не нужно уметь удалять из
очереди — устаревшие события просто гаснут сами.

## Пример прогона

6 сегментов по 4 байта, второй ("EFGH", seq=1005) теряется в сети:

```
[t=0.00] SEND    seq=1001..1021 (все 6, EFGH теряется)

[t=1.00] DELIVER seq=1001 -> Accepted   -> application: "ABCD"
           (ACK отложен — ждём второй full-size сегмент)

[t=1.02] DELIVER seq=1009 -> Accepted (буфер, gap на 1005)
           ACK ack=1005 win=60          <- gap => немедленный ACK
           RTT-сэмпл: 1.02 -> RTO: 10.0 -> 3.06

[t=1.03] DELIVER seq=1013 -> ACK ack=1005 (duplicate #1)
[t=1.04] DELIVER seq=1017 -> ACK ack=1005 (duplicate #2)
[t=1.05] DELIVER seq=1021 -> ACK ack=1005 (duplicate #3!)
           FAST RETRANSMIT (до RTO ещё 2.0 тика)
[t=1.05] RETRANSMIT seq=1005

[t=2.05] DELIVER seq=1005 -> Accepted
           -> application: "EFGH" "IJKL" "MNOP" "QRST" "UVWX"  (каскад, 5 диапазонов)

[t=3.55] ACK ack=1025 win=64   <- финал; Karn's rule: RTT-сэмпл НЕ берём
                                   (среди подтверждённых был ретрансмит)

Final stream: "ABCDEFGHIJKLMNOPQRSTUVWX"
```

Ключевая цифра: **fast retransmit сработал на 2 тика раньше**, чем успел
бы истечь RTO. Это и есть содержательный ответ на "зачем нужен fast
retransmit, если есть таймер" — разница на практике заметна.

## Две ошибки, найденные трассировкой (а не компилятором)

1. Без искусственного сдвига между отправками все не-потерянные сегменты
   планировали приход в один и тот же тик — а `PriorityQueue` не
   гарантирует порядок среди элементов с одинаковым приоритетом.
   Добавлен сдвиг 0.01 на сегмент (реалистично: сериализация пакетов на
   линке не бесплатна).
2. С `DelayedAckMaxDelay=3.0` финальный ACK планировался позже, чем
   успевал перезапуститься RTO-watchdog после fast retransmit — что
   вызывало спонтанный повторный ретрансмит уже доставленных данных.
   Реальный TCP-феномен (spurious retransmission из-за взаимодействия
   delayed ACK и RTO), но не тот, который планировалось показать здесь;
   исправлено уменьшением задержки до 1.5.

## Ограничения

- RTT сэмплируется только один раз в этом прогоне — дальше просто нечего
  сэмплировать (следующий и последний ACK попал под правило Карна).
- Нет congestion window — это глава 4.
- Однонаправленная задержка ACK считается "мгновенной" (упрощение).
