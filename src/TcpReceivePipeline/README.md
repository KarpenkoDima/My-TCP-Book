# TcpReceivePipeline — обзор архитектуры

Упрощённая (но RFC-корректная в заявленных границах) модель одного
направления TCP-соединения: from raw bytes on the wire to a clean
application byte stream, and back — ACK-и, ретрансмиты, congestion control.

Собиралось в четыре захода, каждый — поверх предыдущего, без переписывания
уже проверенного. Этот документ — карта того, что где лежит и зачем,
прежде чем добавлять что-то ещё.

## Зачем это вообще, в одном абзаце

Каждый файл в `Core/` отвечает ровно за один вопрос, который реальный TCP
обязан решать:

| Вопрос | Класс | RFC |
|---|---|---|
| Какие байты уже пришли непрерывно? | `TcpStreamReassembler` | 9293 §3.4 |
| Что реально можно рекламировать как RCV.WND? | `TcpReceiveEndpoint` | 9293 §3.1 |
| Сколько мы отправили и что подтверждено? | `TcpSendWindow` | 9293 §3.3.1 |
| Когда считать сегмент потерянным? | `RetransmissionTimer` + `RetransmissionController` | 6298 |
| Когда ACK-ить сразу, а когда можно подождать? | `DelayedAckPolicy` | 1122 §4.2.3.2, 5681 §3.2 |
| С какой скоростью вообще можно слать данные? | `CongestionController` | 5681 |

Ни один из этих классов не знает о существовании остальных пяти напрямую —
они соединяются только в `Host/Program.cs`, через явные вызовы. Это
осознанное решение: каждый вопрос выше имеет свой, отдельный от других,
жизненный цикл состояния, и смешивание их в одном объекте — ровно тот
класс багов, который эта архитектура должна исключать (см. "Урок №1" ниже).

## Карта файлов

```
TcpReceivePipeline/
├── Core/
│   ├── TcpSequence.cs            — арифметика seq-space по модулю 2^32 (сравнения, Distance)
│   ├── TcpFlags.cs                — биты заголовка TCP (RFC 9293 §3.1)
│   ├── TcpSegment.cs               — readonly ref struct, zero-alloc вид на один сегмент
│   ├── BufferedSegment.cs          — один сохранённый [Start,End) диапазон, ArrayPool<byte>
│   ├── TcpStreamReassembler.cs     — OOO-сборка байтового потока, дедупликация, overlap-trim
│   ├── TcpReceiveEndpoint.cs       — reassembler + честный RCV.WND (ёмкость минус занятое)
│   ├── TcpSendWindow.cs            — SND.UNA/NXT/WND, usable window
│   ├── InFlightSegment.cs          — один неподтверждённый отправленный сегмент (для ретрансмита/RTT)
│   ├── RetransmissionTimer.cs      — SRTT/RTTVAR/RTO по Джейкобсону, backoff по Карну
│   ├── RetransmissionController.cs — очередь in-flight, duplicate-ACK счётчик, Karn's rule
│   ├── DelayedAckPolicy.cs         — когда ACK-ить сразу vs откладывать
│   └── CongestionController.cs     — cwnd/ssthresh: slow start, congestion avoidance, fast recovery
├── Infrastructure/
│   ├── TcpHeaderParser.cs          — zero-alloc парсинг заголовка из ReadOnlySpan<byte>
│   └── SyntheticSegmentWriter.cs   — генерация wire-байт для демо (не для продакшена)
├── Host/
│   ├── Program.WindowAndReassembly.cs  — раунд 1: sliding window + OOO reassembly, без времени
│   ├── Program.DelayedAckAndRto.cs     — раунд 2: + virtual clock, delayed ACK, RTO, fast retransmit
│   └── Program.cs                      — раунд 3 (текущий): + congestion control (RFC 5681)
└── TcpReceivePipeline.csproj
```

Три файла в `Host/` — не мусор и не забытые черновики. Это осознанно
сохранённые снапшоты (по твоему же правилу для MyChannels: снапшот перед
перезаписью), каждый — рабочая, самодостаточная демонстрация своего слоя.
Два ранних снапшота исключены из компиляции в `.csproj`, потому что каждый
содержит собственный `Main()`. Чтобы запустить конкретный этап, временно
замените им `Host/Program.cs` либо измените список `Compile Remove` в проекте.

## Как менялась модель по раундам

**Раунд 1 (`Program.WindowAndReassembly.cs`).** Нет понятия времени —
раунды по принципу "отправитель шлёт, пока есть usable window, затем сеть
доставляет один сегмент, затем ACK". `TcpSendWindow` и старый (уже
переписанный) `SlidingWindowTracker` для приёмной стороны. Здесь же
разделили класс на sender-only (`TcpSendWindow`) и полноценный
receive-side (`TcpReceiveEndpoint`), когда стало ясно, что RCV.NXT не
может одновременно быть полем трекера и производной reassembler'а.

**Раунд 2 (`Program.DelayedAckAndRto.cs`).** Появляется первое настоящее
время: `PriorityQueue<Action,double>` как очередь событий, виртуальный тик
вместо раундов. Отсюда — **паттерн generation-счётчика**, который дальше
используется everywhere: `PriorityQueue` не умеет удалять произвольный
элемент, поэтому вместо отмены события мы инкрементируем счётчик при
каждом логическом "перезапуске" таймера, а сработавший callback сверяет
захваченное им число с текущим. Если не совпало — no-op. Этим способом
устаревают: RTO watchdog (`RetransmissionController.TimerGeneration`) и
отложенный ACK (`DelayedAckPolicy.Generation`) — независимо друг от друга,
у каждого свой счётчик.

**Раунд 3 (`Program.cs`, текущий).** `CongestionController` встраивается
в `TrySendMore()`: usable window становится `min(cwnd, rwnd) - inFlight`
вместо просто `rwnd`. Это не косметика — initial window (16 байт при
MSS=4) реально не даёт отправить больше 4 сегментов сразу, и два хвостовых
сегмента в трассировке уходят на добрую секунду позже, чем в раунде 2.

## Уроки, которые стоило зафиксировать явно

1. **RCV.NXT — одна переменная, а не две.** Первая версия держала RCV.NXT
   и в `SlidingWindowTracker`, и (по факту) в reassembler'е. Разделение на
   `TcpSendWindow`/`TcpReceiveEndpoint` устранило это до того, как оно
   стало багом, а не после.
2. **Generation вместо удаления из очереди событий.** См. выше — сквозной
   паттерн с раунда 2.
3. **Порядок событий в `PriorityQueue` с одинаковым приоритетом не
   гарантирован.** Раунд 2 сначала планировал 5 приходов сегментов в один
   и тот же тик — пришлось ввести искусственный (но реалистичный: время
   сериализации пакета на линке) сдвиг между отправками.
4. **Delayed ACK и RTO могут гоняться друг за другом.** В раунде 2 с
   `DelayedAckMaxDelay=3.0` перезапущенный после fast retransmit RTO-таймер
   срабатывал раньше отложенного ACK и вызывал спонтанный повторный
   ретрансмит уже доставленных данных — реальный TCP-феномен, но не тот,
   который планировался показать. Поправлено уменьшением задержки.
5. **Karn's rule — это "пропустить сэмпл", а не "пропустить сегмент".**
   Если среди подтверждённых одним ACK сегментов есть хоть один
   ретранслированный, RTT-выборка отбрасывается целиком для этого ACK, а
   не только для проблемного сегмента — какому именно из них соответствует
   this ACK, principally неизвестно.

## Что провалидировано, а что нет

Компилятора в песочнице нет — .NET SDK не установлен. Всё, что описано
выше как "проверено", означает: пошаговая ручная трассировка каждого
сценария (арифметика, порядок событий, содержимое буферов) с explicit
проверкой финального результата (`Final stream` совпадает с исходным
payload побайтово). Это не заменяет юнит-тесты и тем более не заменяет
запуск на реальном трафике MikroTik — который остаётся конечной целью
всего NetPulse-проекта.

Не проверено вообще:
- компиляция (наиболее вероятный источник тривиальных ошибок — опечатки,
  пропущенные `using`, несовпадения сигнатур);
- конкурентный доступ (все классы here рассчитаны на один поток/один flow,
  как и заявлено в докстрингах);
- поведение на реальных, а не синтетических, сегментах.

## Накопленные ограничения (актуально на конец раунда 3)

- Нет SYN/FIN state machine — `initialReceiveSequence`/`initialSendSeq`
  готовятся вручную как `ISN + 1`.
- Нет 4-tuple/демультиплексирования — один объект = один уже
  установленный flow.
- Нет checksum, Window Scale, SACK, PAWS/Timestamps.
- `OnDuplicateAckDuringRecovery` (RFC 5681 §3.2 шаг 3) в демо ни разу не
  выполнился — сценарий не породил подходящих условий.
- NewReno partial ACK (RFC 6582) не реализован.
- Нет congestion control pacing — `TrySendMore` выпускает весь доступный
  кусок окна одним циклом.
- `ScheduleWatchdog()` иногда планирует избыточные копии одного и того же
  события (гасятся generation-проверкой, но впустую занимают память
  очереди событий).
